Cylc

Suite Engine

Cylc ("silk") is a suite engine and meta-scheduler that specializes in suites of cycling tasks for weather forecasting, climate modeling, and related processing (it can also be used for one-off workflows of non-cycling tasks, which is a simpler problem). Cylc is free software under the GNU GPL v3 license.

Overview

Cylc's suite design paradigm is simple and intuitive: configure task scheduling using a dependency graph, then configure task runtime properties in an efficient inheritance hierarchy. This suite definition, for example, defines a suite with this dependency graph.

Cylc automatically interleaves tasks from multiple forecast cycles, dependencies allowing. Suites seamlessly transition between delayed and real time operation, they flow around delayed or failed tasks, and they adapt to insertion and removal of tasks at run time.

Who's Using Cylc So Far?

Cylc was originally developed by Dr Hilary James Oliver at NIWA (the National Institute of Water and Atmospheric Research, New Zealand). It is now developed collaboratively by NIWA, the UK Met Office, and others:

  • NIWA - cylc is used with Rose
  • Met Office (UK) - cylc is used with Rose
  • NRL Marine Meteorology Division (USA)
  • Max Planck Institute for Meteorology (Germany)
  • Bureau of Meteorology (Australia) - cylc is used with Rose
  • GFDL (NOAA/Princeton University, USA)

Features

First, see the Overview above.

  • Unified command line interface: cylc COMMAND [OPTIONS] [ARGS].
  • Multi-view graphical user interface with collapsing task families.
  • Clean separation of task scheduling and runtime configuration.
  • Efficient configuration of task runtime properties by multiple inheritance.
  • Distributed suites - submit different tasks by different methods to different hosts.
  • Remote control - interact with suites running on other hosts.
  • task cycling - hours-of-the-day; and stepped daily, monthly, and yearly.
  • Handles several thousand tasks in a single suite.
  • Inter-suite dependence - triggering tasks off tasks in other suites.
  • Suite validation - catch many errors prior to run time.
  • Suite cold-start, warm-start, and restart.
  • Conditional triggering.
  • Task family triggering.
  • Automated failure recovery.
  • Task and suite event hooks and timeouts for central alerting.
  • Simulation and dumy modes - get the scheduling right without running real tasks.
  • Supports the Jinja2 Template Processor in suite definitions.
  • Internal Queues to limit the number of simultaneously active tasks.
  • Task Poll and Kill for supported job submission methods.

Requirements

  • Operating System: Linux or Unix
  • Python 2-series, version: 2.5 or later.x as yet).
  • PyGTK, a Python wrapper for the GTK+ graphical user interface toolkit, included in most Linux Distributions.
  • Pyro 3 (Python Remote Objects) - latest version 3.16.
  • The graphviz graph layout engine (latest version tested: 2.28.0).
  • Pygraphviz, a python interface to graphviz (latest version tested: 1.1).
  • Jinja2, a template processor for Python.
  • LaTeX (for pdflatex) and tex4ht (for htlatex) - needed to generate PDF and HTML copies of the Cylc User Guide from the LaTeX source files.

Cylc has also absorbed xdot (LGPL license) in modified form (no need to install separately).

Documentation

Cylc is documented by command line help, GUI help menus, and the Cylc User Guide.

The latest version of the user guide is held online in HTML format:

  • Cylc User Guide (HTML multi-page)
  • Cylc User Guide (HTML single-page)

These documents, and a PDF version, can also be generated in your local cylc installation by running make in the source tree. Local documentation can be accessed by running the cylc doc command.

Development

Cylc uses the powerful Git distributed source code management system. Git makes branching and merging, forking/cloning, and pushing and pulling changes between repositories very easy. The cylc repository is hosted on GitHub.

We use the integrator or fork and pull model of distributed development. Developers fork the master repository on GitHub, and clone their fork to their local workstation. Changes are pushed to the developer fork and a pull request issued to the master repository, where the core cylc developers review and merge them.
