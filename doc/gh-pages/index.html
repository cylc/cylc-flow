Cylc

Suite Engine

Cylc ("silk") is a suite engine and meta-scheduler that specializes in suites of cycling tasks for weather and climate forecasting and related processing (it can also be used for one-off workflows of non-cycling tasks, which is a simpler problem). Cylc was released by NIWA under the GNU General Public License v3 in 2011.

.

Cylc was selected as a finalist in two categories for the 2012 New Zealand Open Source Awards.

Overview

Cylc's suite design paradigm is simple and intuitive: "draw" a dependency graph in concise textual form, then configure the runtime properties of each task in an efficient inheritance hierarchy. This simple suite definition, for example, defines a suite with this dependency graph. Large suites like this are no more difficult, they just have more tasks.

Cylc suites can interleave tasks from multiple forecast cycles for fast catch up from delays, fast historical case studies, and fast offset parallel trials; they seamlessly transition between delayed and real time operation; they flow around delayed or failed tasks (which will catch up as quickly as possible once the problem is addressed), and they adapt to insertion and removal of tasks at run time.

Cylc's lead developer is Dr Hilary James Oliver of NIWA (the National Institute of Water and Atmospheric Research, New Zealand).

Who Is Using Cylc So Far?

Cylc is (or is about to be) the official suite engine of the following institutions:

  • NIWA - cylc has controlled NIWA's research and forecasting suites since 2010.
  • Met Office (UK) - cylc is used in combination with Rose (https://github.com/metomi/rose).

Cylc is also used by small groups at the following institutions:

  • NRL Marine Meteorology Division (USA)
  • Max Planck Institute for Meteorology (Germany)
  • Bureau of Meteorology (Australia)

Features

First, see the Overview above.

  • Self-documenting command line and graphical interfaces covering all functionality.
  • A multi-view suite control GUI, with collapsible task families.
  • Most tasks require no modification for use with cylc.
  • Clean separation of task scheduling and runtime configuration.
  • Configuration of task runtime properties (environment etc.) via an inheritance hierarchy of nested task families.
  • Remote control: from your desktop box, control a suite running on a remote server that submits tasks to other remote hosts.
  • Extensible cycling: currently hours-of-the-day, and stepped daily, monthly, and yearly cycling.
  • Cylc can handle hundreds of tasks in a single suite (beyond this the suite control GUI may struggle, but you can run very large suites via the command line tools).
  • Suites can be distributed across multiple hosts on a heterogeneous network.
  • Supports use of many job submission methods within a suite (e.g. loadleveler, pbs, sge, background, at)
  • Suites are validated against a spec to catch many errors prior to run time.
  • Suite security: access to running suites is automatically passphrase authenticated.
  • Submit single tasks exactly as the suite would (including job submission method and execution environment).
  • Suites are accessed via hierarchical names (which to group and organize suites) registered in a private suite database .
  • A lockserver can be used to prevent accidental invocation of multiple instances of a suite or task at the same time (even for distributed suites).
  • Suites cold-started, warm-started, and stopped and restarted in any state of operation.
  • Explicit cold start bootstrapping to get warm-cycling suites up and running.
  • Recursive purge: remove sub-trees of tasks from a running suite through to some future cycle.
  • General conditional triggers.
  • Task family triggers.
  • Automated task failure recovery (several methods supported).
  • Task and suite event hooks for central alerting and timeouts.
  • Portable suites are easy to construct - just copy another user's suite and run it.
  • Simulation mode: get the scheduling right, on an accelerated clock, without running real tasks.
  • One-off asynchronous tasks for constructing one-off suites of non-cycling tasks.
  • Repeating asynchronous tasks for parallel processing of real time satellite datasets.
  • Jinja2 Template Processor supported, for programmatic generation of suite definitions.
  • Internal Queues - limit the number of active (submitted or running) tasks.

Requirements

  • Operating System: Linux or Unix
  • Python Version: 2.4 or later, but not Python 3.x as yet. 2
  • PyGTK, a Python wrapper for the GTK+ graphical user interface toolkit, included in most Linux Distributions.
  • Pyro 3 (Python Remote Objects) - latest version tested 3.14. 3
  • The graphviz graph layout engine (latest version tested: 2.28.0).
  • Pygraphviz, a python interface to graphviz (latest version tested: 1.1).
  • Jinja2, a template processor for Python.
  • LaTeX (for pdflatex) and tex4ht (for htlatex) - needed to generate PDF and HTML copies of the Cylc User Guide from the LaTeX source files.

Python and Pyro are essential. PyGTK is required by the gcylc GUI (but you can control and monitor cylc suites from the command line). Graphviz and pygraphviz are required for dependency graphing and the graph-based suite control GUI (but you can also run cylc without them). Jinja2 can be used for advanced programmatic generation of suite definitions.

Cylc has absorbed the following software in slightly modified form (you do not need to install them separately):

  • xdot, a graph viewer (LGPL license)
  • the ConfigObj and Validate Python modules (BSD license)

Documentation

Cylc is documented by command line help, GUI help menus, and the Cylc User Guide.

The latest version of the user guide is held online in HTML format:

  • Cylc User Guide (HTML multi-page)
  • Cylc User Guide (HTML single-page)

Development

Cylc uses the powerful Git distributed source code management system. Git makes branching and merging, forking/cloning, and pushing and pulling changes between repositories very easy. The cylc repository is hosted on GitHub.

We use the integrator or fork and pull model of distributed development. Developers fork the master repository on GitHub, and clone their fork to their local workstation. Changes are pushed to the developer fork and a pull request issued to the master repository maintainer (Hilary Oliver), who reviews the changes and pulls them into the master repository.
